# バッチ追加計画

## batch-00-sample
- 目的: 青山繁晴の完成見本1名で、データ構造、UI、検証スクリプトの基準を固定する
- 件数: 1名
- 状態: 完了
- 完了条件:
  - JSON と写真が紐づく
  - 出典 PDF とページ番号が記録されている
  - 単語帳 UI が成立する
  - validate が通る

## batch-01-20
- 目的: 青山繁晴の完成見本を基準に、20名で表記ルールと運用ルールを検証する
- 件数: 20名（完成見本1名 + 追加19名）
- 状態: 完了
- 進め方:
  - 完成見本 `hr-0001` 青山繁晴を基準カードにする
  - 実データ配分は g01=8名、g02=4名、g03=8名、g04=0名で固定する
  - g01 小選挙区基本型から着手し、g02、g03、g04 の順で広げる
  - 政党、選出区分、都道府県、比例ブロックの表記をそろえる
  - 各人に写真と `sourcePdf` `sourcePage` を付ける
  - 追加のたびに validate を回す
  - UI 一覧と絞り込みで破綻がないか確認する
- 完了条件:
  - 表記ゆれの修正方針が固まる
  - 画像配置と JSON 命名規則が崩れない
  - 完成見本を見ずに同じ粒度で追加できる
  - 青山繁晴基準のグループ順で追加しても差分運用が破綻しない

## batch-02-50
- 目的: 20名検証で固まったルールを使って、50名追加の標準手順を確立する
- 件数: 50名
- 状態: 実名、sourcePage、写真、実作業ファイル、50名全員の draft JSON まで投入済み。`hr-0060` までは source 確認と `career` 反映まで完了
- 前提:
  - batch-01-20 でルール固定済み
  - バリデーション項目が不足なく機能している
  - batch-02-50 の groups / roster / worklist / review が配置済み
  - 50枠は g01=19, g02=10, g03=19, g04=2 に配分済み
  - 基本項目は 50名全件で `draft` 入力済み
- 完了条件:
  - 追加フローが機械的に繰り返せる
  - レビュー時に差分追跡しやすい
  - groups / roster / worklist / review の4点セットで再開できる
  - `career` 追記と UI/目視確認を経て `draft` から `verified` へ上げる導線が崩れない
